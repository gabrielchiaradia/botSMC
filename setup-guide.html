02 Subir el proyecto a GitHub
Descomprimir el zip y entrar al directorio 
unzip smc-bot-dashboard.zip
cd smc-bot
Crear repositorio en GitHub

Ir a github.com/new → nombre: smc-bot → sin inicializar con README ni .gitignore → Create repository.

Inicializar git y hacer el primer push 
git init
git branch -M main
git remote add origin https://github.com/TU_USUARIO/smc-bot.git
git add .
git commit -m "feat: SMC bot Fases 1-4 completo"
git push -u origin main
Verificar que no subiste credenciales 
# Debe devolver vacío. Si muestra algo, revisá antes de pushear.
git log --oneline -1 | xargs git show | grep -i "api_key\|secret\|token"
Importante El archivo .env nunca debe aparecer en el repositorio. El .gitignore ya lo excluye, pero siempre es bueno verificar.
03 Crear el VPS en Vultr

Vultr Tokyo es la ubicación óptima porque los servidores de Binance Futures están en AWS Tokyo. La latencia desde Tokyo es ~0.6ms vs ~50ms desde Latinoamérica.

Generar SSH key (si no tenés una) 
ssh-keygen -t ed25519 -C "smc-bot-vps"
# Guardar en ~/.ssh/id_ed25519 (Enter para todo)
# Ver la clave pública que vas a pegar en Vultr:
cat ~/.ssh/id_ed25519.pub

04 Setup automático del servidor

Un solo comando instala Docker, configura el firewall, crea el usuario botuser, clona el repositorio y habilita el servicio para que arranque al reboot.

# Ejecutar como root en el VPS
REPO_URL=https://github.com/TU_USUARIO/smc-bot.git \
  bash <(curl -fsSL https://raw.githubusercontent.com/TU_USUARIO/smc-bot/main/deploy/setup_vps.sh)

Qué hace el script

AcciónDetalle
Actualizar sistemaapt update && apt upgrade
Crear usuario botuserSin acceso root, solo puede usar Docker
Configurar firewallSolo puerto 22 (SSH) abierto. Todo lo demás bloqueado.
Instalar fail2banBloquea IPs que intenten fuerza bruta en SSH
Instalar Docker CEVersión oficial, no la del apt por defecto
Clonar repositorioEn /home/botuser/smc-bot
Crear .envCopiado desde .env.example, listo para completar
Servicio systemddocker compose up automático al reboot
El SSH queda igual El script abre explícitamente el puerto 22 antes de activar el firewall. No te quedás sin acceso.

Cómo obtener las API keys de Binance Testnet

Crear cuenta en el Testnet

Ir a testnet.binancefuture.com → registrarse con GitHub → ir a Settings → API Key → generar.

Agregar la IP del VPS a la whitelist

En la configuración de la API key en Binance, agregar la IP del VPS en "Restrict access to trusted IPs only". Sin este paso Binance rechazará las órdenes aunque la key sea correcta.

# Ver la IP pública del VPS desde el VPS:
curl ifconfig.me

Cómo obtener el Telegram Chat ID

# 1. Crear bot en @BotFather → /newbot → copiar el TOKEN
# 2. Enviar cualquier mensaje al bot
# 3. Abrir esta URL en el navegador (reemplazar TOKEN):
https://api.telegram.org/botTOKEN/getUpdates
# 4. Buscar "chat":{"id": XXXXXXXX} → ese número es el CHAT_ID
06 Iniciar el bot con Docker 
cd ~/smc-bot

# Build + arranque en background
docker compose up -d --build

# Verificar que está corriendo
docker compose ps

# Ver logs en tiempo real (Ctrl+C para salir sin detener)
docker compose logs -f smc-bot
Qué esperar en los logs Vas a ver líneas como: Precio: $64000 | Tendencia: ALCISTA | Sesión: london | Score: 30/100 y Sin señal. Mercado lateral. Eso es normal — el bot analiza y loguea cada ciclo aunque no opere.

10 Actualizar el código desde Git

Cada vez que hacés cambios en tu máquina y los subís a GitHub:

# En tu máquina local:
git add .
git commit -m "fix: ajuste de parámetros"
git push origin main

# En el VPS:
bash ~/smc-bot/deploy/update.sh              # Pull + rebuild + restart
bash ~/smc-bot/deploy/update.sh --no-build   # Solo pull + restart (sin rebuild)
Cuándo usar --no-build Si solo cambiaste parámetros en .env, lógica de estrategia, o scripts Python (sin tocar requirements.txt), el flag --no-build es más rápido porque no reinstala las dependencias.
11 Pasar a dinero real
Semanas 1-2
Entorno funcionando
VPS corriendo, bot en paper trading, logs fluyendo. Verificar que las señales tienen sentido mirando los gráficos.
Semanas 3-6
Paper trading activo — 30 días mínimo
No se puede acelerar. Observar win rate real vs backtest, sesiones donde funciona mejor, señales descartadas.
Semanas 7-8
Análisis y ajuste de parámetros
Revisar journal de señales. Ajustar uno o dos parámetros si algo no funciona. Correr backtest para confirmar.
Semana 9
Live en Testnet con órdenes reales
Cambiar a --live con TESTNET=true. Verificar que las órdenes, el SL y el TP se ejecutan correctamente.
Semana 10+
Real con capital mínimo
Cambiar TESTNET=false. Arrancar con $50-100 USDT y apalancamiento 2x máximo. Nunca arriesgar más del 1% por trade.
Para pasar a real Cambiar en .env: TESTNET=false y LEVERAGE=2. Crear nuevas API keys en Binance.com real (no testnet) con la IP del VPS en la whitelist. Agregar fondos reales a la cuenta de Futuros.

13 Perfiles de estrategia

Los perfiles son capas de filtros adicionales que se aplican después de la señal SMC base. Permiten probar distintas combinaciones sin modificar el código core.

Perfiles disponibles

NombreQué hace
baseEstrategia SMC pura: FVG + OB + BOS. Sin filtros extra. Sirve como baseline.
ob_bosSolo acepta Order Blocks que fueron precedidos por un BOS. Reduce cantidad pero mejora calidad.
ema_filterFiltra con EMA 50/200 en HTF + distancia al precio + vela de confirmación (engulfing/martillo).
ob_bos_emaCombina OB+BOS con los filtros EMA. El más restrictivo.

Uso en backtest

# Un solo perfil
python scripts/run_backtest.py --strategy ob_bos

# Comparar todos los perfiles
python scripts/run_backtest.py --compare

# Comparar perfiles específicos
python scripts/run_backtest.py --compare --profiles base ob_bos

# Listar perfiles disponibles
python scripts/run_backtest.py --list-profiles

Crear un perfil nuevo

  1. Crear strategy/profiles/profile_mi_perfil.py
  2. Heredar de BaseProfile e implementar nombre, descripcion y filtros()
  3. Registrarlo en strategy/profiles/__init__.py
  4. Listo — el backtest y el bot lo reconocen automáticamente
14 Trailing dinámico (TP/SL adaptativo)

En lugar de un TP fijo en 2R, el sistema puede mover el SL y el TP dinámicamente a medida que el precio avanza a favor. Hay 4 modos disponibles:

Modos de trailing

ModoCómo funcionaIdeal para
none TP y SL fijos. Comportamiento original (RR 1:2 estático). Baseline, mercados laterales
escalones Cada vez que el precio avanza 1R a favor, el SL sube 1R. No hay TP fijo: el trade corre hasta que el trailing SL lo cierre. Puede capturar 3R, 4R, 5R+. Tendencias fuertes
atr El SL sigue al mejor precio a una distancia de N×ATR (default 2×ATR). Sin TP fijo. Se adapta a la volatilidad en tiempo real. Movimientos explosivos con volatilidad variable
hibrido Cierra 50% de la posición al TP original (2R), asegurando ganancia. El 50% restante corre con trailing por escalones sin límite. Mejor de ambos mundos. Uso general recomendado

Ejemplo visual: modo escalones (LONG)

Entrada $100, SL original $98, dist 1R = $2

Precio llega a $102  →  SL sube a $100 (breakeven)
Precio llega a $104  →  SL sube a $102 (+1R asegurado)
Precio llega a $106  →  SL sube a $104 (+2R asegurado)
Precio revierte a $104  →  cierra en $104 = ganancia +2R

Con TP fijo en 2R habría cerrado en $104 igual,
pero si el precio seguía a $110 el trailing captura +4R.

Uso en backtest

# Un trailing específico
python scripts/run_backtest.py --trailing escalones

# Combinar perfil + trailing
python scripts/run_backtest.py --strategy ob_bos --trailing hibrido

# Comparar los 4 modos de trailing con el mismo perfil
python scripts/run_backtest.py --compare-trailing
python scripts/run_backtest.py --compare-trailing --strategy ob_bos

# Comparar perfiles con un trailing fijo
python scripts/run_backtest.py --compare --trailing hibrido

# Ajustar multiplicador ATR para modo atr
python scripts/run_backtest.py --trailing atr --atr-mult 2.5

# Ajustar cooldown post-SL (velas de espera)
python scripts/run_backtest.py --trailing escalones --cooldown 5

Métricas nuevas en el reporte

MétricaQué mide
Avg RR RealEl RR promedio que realmente se logró (no el teórico 1:2)
Max R CapturadoEl máximo R que logró un trade individual
trailing_modeQué modo se usó (aparece en cada trade del JSON)
Nota: El trailing dinámico actualmente funciona solo en el backtest. En el bot en vivo (run_bot.py) las órdenes siguen siendo TP/SL fijos ejecutados por Binance como OCO. Una vez que identifiques el mejor modo en backtest, se puede implementar en producción.
15 Mejoras v2 del motor de backtest

El motor de backtest incluye varias mejoras respecto a la versión original para producir resultados más realistas:

Correcciones

MejoraProblema originalFix
BOS por posición El slicing de swings usaba labels del índice en vez de posición numérica, pudiendo incluir swings fuera de rango. Cambiado a iloc para slicing correcto.
Resolución TP/SL conservadora Si una vela tocaba TP y SL, siempre se priorizaba TP (sesgo optimista). Se asume SL primero cuando ambos se tocan en la misma vela.
Comisiones por tramo Comisión calculada como precio_cierre × 2 para ambos tramos. Entrada sobre precio_entrada, salida sobre precio_cierre.
Sharpe por timeframe Se usaba sqrt(252) (datos diarios) para velas de 15 minutos. Anualización correcta según barras/año del timeframe.
Mitigación de FVGs FVGs ya tocados seguían contando como confluencia válida. Se marcan como mitigados y se excluyen de señales futuras.

Nuevas funcionalidades

FeatureDescripciónDefault
Cooldown post-SL Después de un stop loss, espera N velas antes de la siguiente entrada. Evita rachas de pérdidas en mercados laterales. --cooldown 3
Límite de pérdida diaria Si la pérdida acumulada del día supera el umbral, no se abren nuevas posiciones hasta el día siguiente. MAX_DAILY_LOSS=0.03 (3%)
Filtro de volatilidad mínima Descarta señales si el ATR es menor al 0.1% del precio (mercado comprimido, stops muy ajustados). Siempre activo
Trailing dinámico 4 modos de trailing TP/SL. Ver sección 14. --trailing none
